## Sprint Conventions

**Cadence**: Monday = plan sprint; Friday = mark completions, begin next sprint planning.

**Statuses**:
- Done — completed this sprint
- In Progress — active (should be finalized by sprint end)
- Not Completed — failed to finish; team decides Monday: retry (new task) or drop
- Won't Do — intentionally dropped
- Blocked → Not Completed if still blocked at sprint end

**No carryover**: Incomplete work is never moved forward. If retried, a new task is created in the current sprint.

**Backlog** = items with no sprint assigned (candidates, not committed).

**Planning assumption**: The current sprint plan is the output of Monday planning, which already incorporates retry decisions for any Not Completed items from last sprint. Treat the current sprint task list as the team's deliberate, reviewed commitment. Last sprint context is useful for understanding *why* something was retried and whether the retry is better-scoped — not for adding risk on top of what the team planned.

**Risk signals**:
- "Not Completed" items from last sprint = failed work, not in-progress work
- "In Progress" at planning time = edge case (not yet finalized, not a commitment)
- Never interpret "Not Completed" without a sprint filter — old data will surface
- A missing retry is a gap worth flagging: if something clearly needed a follow-up but no corresponding task exists in the current sprint, that's a risk
